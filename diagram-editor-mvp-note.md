# 流程图编辑方案草稿

## 目标

先验证这一件事：

`代码式源文件 + AI 修改 + 可视化编辑 + 稳定同步`

第一版优先：

- 做成 VS Code 插件
- 图文件放在项目里
- 打开文件后，右侧可预览、可基础编辑

不先做：

- 独立软件
- 专业级画图器
- 手工调线、调锚点、调拐点

## 推荐方向

先做：

`语义文件 + 布局文件`

也就是：

- 源文件只存“节点、文字、连线”
- 布局文件只存“位置、尺寸”

## 文件结构

```text
example.flow
example.flow.layout.json
```

`example.flow`

```text
dir LR

node start "开始"
node review "审批"
node done "完成"

edge start -> review
edge review -> done
```

`example.flow.layout.json`

```json
{
  "nodes": {
    "start": { "x": 80, "y": 120, "w": 140, "h": 56 },
    "review": { "x": 320, "y": 120, "w": 140, "h": 56 },
    "done": { "x": 560, "y": 120, "w": 140, "h": 56 }
  }
}
```

规则：

- AI 主要改 `example.flow`
- 用户拖拽主要改 `layout.json`

## 为什么不直接用纯 Mermaid

Mermaid 适合渲染，不太适合回写编辑。

问题在于：

- 语法糖多
- 节点常常是隐式创建
- 稳定 id 不够强
- 可视化改完后，反写源码容易脆

所以第一版更适合：

- 长得像 Mermaid
- 但语法更受控

## 插件形态

```text
+-------------------+---------------------------+
| 左边：代码        | 右边：预览 / 编辑        |
|                   |                           |
| node start ...    |   [开始] -> [审批]       |
| edge start -> ... |                \          |
|                   |                 -> [完成] |
+-------------------+---------------------------+
```

第一版支持：

- 拖动节点
- 改节点文字
- 新增/删除节点
- 新增/删除连线
- 一键自动整理布局

第一版不支持：

- 手工调线形状
- 手工调锚点
- 细致样式编辑

## 内部结构

```text
源码
  |
  v
解析器
  |
  v
图模型  <---->  布局模型
  |
  v
渲染器
  |
  v
可编辑画布
```

图模型只关心：

- node: `id` `label`
- edge: `from` `to` `label?`

布局模型只关心：

- `x` `y` `w` `h`

关键点：

不要把编辑器直接绑死在 Mermaid 上。  
内部应该有自己的图模型，这样以后要换成 Mermaid 子集或自定义 DSL 都可以。

## 编辑流程

### AI 改图

```text
AI 改源码
-> 重新解析
-> 老节点尽量保留原布局
-> 新节点自动放置
-> 重新渲染
```

### 用户拖拽改图

```text
拖拽 / 改字 / 连线
-> 更新图模型或布局模型
-> 写回源码或布局文件
-> 重新渲染
```

规则：

- 结构变化，写回源文件
- 位置变化，写回布局文件

## 连线策略

第一版目标：

- 连线全自动
- 默认尽量规整
- 拖完后自动重算
- 提供“整理布局”按钮

第一版不追求：

- 用户随便拖完，系统仍然永远像 Mermaid 一样完美排版

这个目标太重，不适合第一版。

## MVP 成功标准

- AI 能稳定生成和修改源文件
- 用户能完成基础可视化编辑
- 多次“AI 改 + 人拖拽”后，图不坏
- 默认布局比较整齐
- 拖拽后连线还能接受

## 结论

第一版最合理的路线是：

`VS Code 插件 + 受控 DSL + layout sidecar + 自动连线`

先把“好不好用”验证出来，再决定后面是：

- 继续兼容 Mermaid 风格
- 还是正式走自定义 DSL
