# 防踩坑

这个文件记录项目里已经踩过、以后大概率还会再踩一次的坑。

## Custom Editor

### 1. VS Code 注册 API 不要写错

- `.flow` 自定义编辑器要用的是 `vscode.window.registerCustomEditorProvider(...)`
- `provider` 对象里实现的是 `resolveCustomTextEditor(...)`
- 不要写成 `registerCustomTextEditorProvider(...)`

错误表现：

- Extension Host 激活失败
- 报错类似：

```text
vscode.window.registerCustomTextEditorProvider is not a function
```

### 2. 文档同步链路里，`isApplyingHostEdit` 是底线保护

在 Custom Editor 这条链路里，最容易出问题的是：

1. webview 改图
2. 宿主写回 `.flow`
3. `onDidChangeTextDocument` 被触发
4. 又反过来刷新自己
5. 造成重复刷新、抖动，严重时可能形成循环

当前的底线保护是：

- 宿主写回前设 `isApplyingHostEdit = true`
- 写回结束后再复位
- `onDidChangeTextDocument` 里先检查这把锁

结论：

- 后续凡是改 `persistGraph`
- 改 `persistSourceText`
- 改 `refreshFromDocument`
- 改 `onDidChangeTextDocument`

都必须先检查这把锁是不是还成立。

### 3. `applyEdit` 失败和异常不是一回事

在 `persistSourceText(...)` 里：

- `await vscode.workspace.applyEdit(edit)` 返回 `false`
- 应该走明确的降级路径
- 不要用 `throw` 去驱动正常控制流

推荐思路：

```js
const applied = await vscode.workspace.applyEdit(edit)
if (applied) {
  await document.save()
  return 'workspaceEdit'
}

await saveDiagramSource(fsLike, sourcePath, sourceText)
return 'fsWrite'
```

### 4. 宿主兜底异常要给用户可见提示

只写 `outputChannel` 不够。

对于 message handler 的非预期异常，至少要补一层：

```js
vscode.window.showErrorMessage(`Diagram editor failed: ${errorMessage}`)
```

否则用户只会看到“没反应”，不知道是宿主炸了。
