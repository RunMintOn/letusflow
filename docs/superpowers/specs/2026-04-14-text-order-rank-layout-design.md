# 文本顺序驱动同层布局 设计

**日期：** 2026-04-14  
**目标：** 保持 `.flow` 文本作为唯一真相，在不引入 sidecar 或拖拽持久化的前提下，让“整理布局”稳定遵循文本声明顺序，避免同层节点被 Dagre 为全局最优而随意换序。

---

## 1. 背景与结论

当前项目已经具备一条部分成立的链路：

- `.flow` 按文本顺序解析为 `graph.nodes`
- Dagre 会读取 `graph.nodes` 顺序并通过 edge weight 受到一定影响
- 因此修改文本顺序后，布局结果通常会跟着变化

但这个顺序信号目前只是 **弱约束**。

在结构对称、长边较多、交叉线博弈明显的图上，Dagre 仍会为了全局更优而拆散局部排布。用户看到的现象就是：

- 文本顺序确实影响了图
- 但同层节点仍可能出现“三个在左，一个在右”这类局部不整齐结果

这次设计的结论是：

- **继续坚持 `Text is Truth`**
- **不做 sidecar，不做拖拽锚点持久化**
- **不引入第二种布局模式**
- **把“整理布局”的语义明确为：**
  - **Dagre 决定 rank（层级）**
  - **`.flow` 文本顺序决定 rank 内顺序**

也就是说，这次不是修改 DSL，而是把“文本顺序”从弱提示提升为更强的布局规则。

---

## 2. 设计目标

### 必须满足

- 用户或 AI 只需要修改 `.flow` 文本顺序，就能稳定影响同层节点排序
- “整理布局”后，同一 rank 内节点按 `.flow` 节点声明顺序排列
- Dagre 仍负责主方向分层，不重写整套布局算法
- 不引入新的持久化文件
- 不要求用户理解“布局约束”或“编辑器状态”

### 明确不做

- 不持久化拖拽后的节点位置
- 不把拖拽结果变成顺序约束
- 不新增 `.layout.json`
- 不扩展 `.flow` DSL 语法
- 不提供多种布局模式让用户切换

---

## 3. 推荐方案与取舍

### 方案 A：Dagre 分层 + rank 内文本顺序后处理

这是推荐方案。

做法：

1. 先运行现有 Dagre，得到基础 layout
2. 根据布局结果识别 rank
3. 对每个 rank 内的节点，按 `.flow` 中的节点声明顺序重新排序
4. 沿 rank 内的交叉轴重新分配坐标，保持统一间距

对于方向：

- `LR`：Dagre 决定 `x` 分层，文本顺序决定同层节点的 `y` 排列
- `TD/TB`：Dagre 决定 `y` 分层，文本顺序决定同层节点的 `x` 排列

优点：

- 不改 DSL
- 不改持久化模型
- AI 改文本即可直接控制排序
- 语义清晰，符合当前产品定位

缺点：

- 会牺牲一部分 Dagre 的全局最优
- 某些图的交叉线可能略有增加

### 方案 B：继续只靠 Dagre 权重

不推荐。

原因：

- 当前实验已经证明它只能提供弱提示
- 一旦遇到结构对称或长边，全局优化仍会覆盖局部顺序
- 用户无法预测结果

### 方案 C：sidecar / 锚点 / 拖拽持久化

不推荐作为当前方向。

原因：

- 会把产品从“代码驱动可视化查看器”推向“半白板编辑器”
- AI 改 `.flow` 与布局偏好同步会显著复杂化
- 明显超出当前最小有价值改造范围

---

## 4. 布局语义定义

### 4.1 单一真相原则

`.flow` 文本仍是唯一真相。

这意味着：

- 节点声明顺序是排序依据
- edge 定义继续只表达连接关系
- 图形视图不引入独立真相

### 4.2 “整理布局”的新定义

“整理布局”不再只是“让 Dagre 自由发挥”，而是：

1. Dagre 负责决定节点属于哪个 rank
2. 系统在每个 rank 内按文本声明顺序重排节点
3. 系统用统一间距把这些节点重新摆齐

这使得“整理布局”成为一个更可预测的动作：

- 它会重新整理
- 但不会在同层里背离文本顺序

### 4.3 rank 识别规则

不要求 Dagre 显式输出 rank 编号。

当前实现可直接基于主轴坐标识别 rank：

- `LR`：同一个 `x` 值视为同一 rank
- `TD/TB`：同一个 `y` 值视为同一 rank

这与当前布局输出是兼容的，且不需要引入额外 Dagre 内部依赖。

### 4.4 rank 内排序规则

对每个 rank：

- 取出属于该 rank 的全部节点
- 按它们在 `graph.nodes` 中出现的顺序排序
- 再沿交叉轴重新放置

排序不依赖边关系、节点 label 或拖拽历史，只依赖文本声明顺序。

### 4.5 rank 内坐标重分配

重排时：

- 保持主轴坐标不变
- 仅重排交叉轴坐标
- 使用当前 spacing 派生出的 `nodesep`
- 节点尺寸仍按现有尺寸规则计算

这样可以最大程度保留 Dagre 的层级结构，只收紧并稳定同层顺序。

---

## 5. 架构影响

### 5.1 DSL 与序列化

`parseDiagram()` 和 `serializeDiagram()` 无需新增语法。

现有行为已经足够：

- parser 保留节点声明顺序
- serializer 按 `graph.nodes` 顺序写回文本

### 5.2 布局层

改动集中在 `layout.js`。

新的布局流程应为：

1. 运行 Dagre 得到基础布局
2. 基于方向识别 rank
3. 对 rank 内节点按 `graph.nodes` 顺序重排
4. 输出最终 layout

为保持实现清晰，建议把 rank 识别与 rank 内重排拆成独立 helper，而不是全部塞在 `autoLayoutGraph()` 里。

### 5.3 编辑器与宿主

宿主、webview、消息协议不需要变化。

因为：

- “整理布局”
- 初始加载
- spacing 变化
- 任何走 `autoLayoutGraph()` 的路径

都会自动获得新的排序语义，无需新增消息或额外状态。

---

## 6. 风险与边界

### 6.1 交叉线可能增加

这是有意接受的取舍。

当前目标是：

- 提高可预测性
- 保证文本顺序可控

而不是继续追求 Dagre 的局部最优交叉数。

### 6.2 只解决同层排序稳定性

这次不承诺：

- rank 本身完全不变
- 全图永远没有大位移
- 节点在不同 rank 之间永不变化

因为 rank 仍然由 Dagre 决定。

这次解决的是：

- **同一 rank 内谁在前、谁在后**

### 6.3 不处理拖拽语义

当前节点拖拽仍然只应被视为临时视图调整。

这次设计不试图让拖拽结果参与布局真相，也不为拖拽增加持久化逻辑。

---

## 7. 测试策略

需要补充的验证重点：

- 同一 rank 内两个或多个节点会按 `graph.nodes` 顺序排列
- `LR` 与 `TD` 两种方向都能稳定遵循文本顺序
- 对称图上，文本顺序变化会稳定影响同层顺序
- 当前已有的层级、尺寸、spacing 测试不应退化

建议新增一类针对“rank 内顺序”的布局测试，而不是只验证主方向坐标大小关系。

---

## 8. 最终结论

如果项目继续坚持“代码驱动可视化查看器”的定位，那么最合理的路线不是锚点系统，也不是 sidecar，而是：

- **保留 `.flow` 作为唯一真相**
- **让文本顺序成为更强的布局规则**
- **把“整理布局”定义成：按文本顺序整理，而不是按 Dagre 喜好重排**

这条路复杂度中等，架构干净，和 AI 工作流天然兼容，是当前最值得推进的方向。
